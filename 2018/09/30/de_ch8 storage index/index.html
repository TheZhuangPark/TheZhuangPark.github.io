Storage and indexing

Data on external storage

Disk: can retrieve random page at fixed cost,reading in consecutive pages is cheaper than reading in random order.

Tape: Can only read consecutive pages.Cheaper than disks,often used for archival storage.

File organization: Record id is sufficient enough to locate record. sorted index allow us to find record id fast with given value in index search key fields.( for example binary search in sort index)

Architecture: Buffer manager stages pages from external storage to main memory buffer pool. File and index layers make calls to buffer manager.

Alternative File Organization

Many alternative File Organizations exit, each ideal for some situation but not so good for in other situation.

Heap(random order)files: suitable for typical access, a file scan retrieving all records.

Sort Files: Best if records must be retrieved in some orders, or range of records is needed(consecutive records)

Indexes: Data structures organize records via tree and hashing. they speed up searches based on search key. But updates are slower, beacuse they need to update both records and index.

indexes

an index on file speed up selection on search key fields for index. any subset of fields of relation can be the search key, search key is not the same as key (mininal set of fields to identify a record uniquely)

Alternatives for Data Entry k in index

3 alternatives
1) data record with key value k
2) (k, rid of data record with search key value k) is like (k, rid)
3) (k, list of rids of record ), is like(k, rid, sid, age)

what you chose to be data entries is orthogonal to indexing technique used to locate data entries ( like B+ tree, hash-based structures, or index contains auxiliary information)

Alternative 1

1 if it is used, index is a file organization for data records.
2 at most one index on given collection data record can use alternative 1 otherwise data record are duplicated, redundant and potentially inconsistant.
3 if data record are large, size of auxiliary information of index is also large.

Alternative 2 and 3

Data entries musch smaller than data records, bettern than alternative 1 with large records. Alternative is more compact than alternative 2, but leads to variable sized data entries.

index classification

1 primary: search key contains primary key.
2 Unique index: search key contains a candidate key
3 clustered: order of data record is the same or close to order of data entries, called clustered index (include alternative 1)
don’t know about file that can be clustered on at most one search key?

cost of search records base on whether index is clustered or not!

clustered vs unclusted index

to build clustered index, first sort heap file where data are stored.
still don’t know which one is better

Hash-Based indexes

1 Good for equality selections, index is a collection of buckets.
2 Bucket is primary page plus zero or more pages.
3 Hashing function h: h(r) = bucket in which record r belongs. h looks at the search key fields of r.

If alternative 1 is used, the buckets contain data records; otherwise, they contain (key,rid), (key,rid-list) pairs.

B+ Tree indexes

Leaf pages contain data entries, and are chained(pre & next), Leaf page is Leaf node.

Non-leaf page contain index entried and tree searches.

Choice of indexes

what indexes should we create ?
1 key field that can be search key.
2 relations should have indexes.
3 how many indexes should be build.
4 for each index, what kind of an index should it be? clustered, hash/tree?

A: consider the most important queries.

B: trade-off indexes can make queries go faster and updates slower. Require disk space, too.

C: select attributes in where clause.
use hash index for exact match condition suggests.
use tree index for range index.
use Clustering for range query and equality query if there are many duplicates.

D: multi-attribute search keys should be considered when a where clause contain several condition.

E: order of attributes is important for range query and sometimes enable index-only strategies for important query. (what is index-only strategies, why is clustering is not important on index-only?)

F: chose indexes that beneft as many query as possible. since only one index can be clustered per relation( why only one?)

Examples of Clustered indexes

ex1: select E.dno FROM Emp E WHERE E.age>40.

first because it’s range query so use clustering and tree index.

ex2: select E.dno,COUNT(*) FROM Emp E WHERE E.age>10 GROUP BY E.dno.

range query, use clustered E.dno.

ex3: select E.dno FROM Emp E Where E.hobby=Stamps.

use hash-index , clustering on E.hobby.

indexes with composite search keys


search key on combination of fields.
to retrieve records with age=30 AND sal=4000 index (age, sal) would be better.

if 20<age<30 AND 3000<sal<5000，clustered tree index on (age,sal)

if age=30 AND 3000<sal<5000, clustered (age,sal)index much better.

index-only plans

mean that query can be answered only by using index without retrieving any tuple from relations.
